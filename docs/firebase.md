# Firebase Channel

The Firebase channel routes C2 traffic through [Firebase Realtime Database](https://firebase.google.com/products/realtime-database) — a cloud-hosted JSON store with a simple REST API. All traffic goes to `firebaseio.com`.

```bash
# Switch to Firebase
CHANNEL=firebase  # in .env
```

---

## Why Firebase

| | Google Sheets | Firebase |
|---|---|---|
| Read | CSV export (polling) | GET JSON (polling) |
| Write | Google Forms POST (append-only) | PUT (overwrite/update) |
| Delete | Not supported — manual cleanup | `DELETE` request |
| Field size limit | ~4000 chars (Forms limit) | 10 MB per node |
| Inbox cleanup | Manual (`sheets_c2_cleanup.gs`) | Automatic after result confirmed |
| Setup | Spreadsheet + 2 Forms + Apps Script | 1 Firebase project, set rules |

The key difference is **DELETE support**. After a result arrives the server automatically removes the corresponding inbox entry — the inbox stays clean without manual intervention. Sheets is append-only by design (Google Forms has no delete endpoint).

---

## Setup

Run the setup wizard and select Firebase when prompted for a channel:

```bash
python setup_wizard.py
```

The wizard walks through:
1. Creating a Firebase project and Realtime Database in the Firebase console
2. Setting database rules to test mode (public read/write — encryption handles content security)
3. Collecting the database URL and paths
4. Running a live connection test (write + delete a test entry)
5. Generating field name obfuscation maps (`FIREBASE_INBOX_COLUMN_MAP` / `FIREBASE_OUTBOX_COLUMN_MAP`) — enabled by default; reuses Sheets maps if already generated in Step 2
6. Writing all env vars to `.env`

### Manual setup

1. Go to [console.firebase.google.com](https://console.firebase.google.com)
2. Create a new project
3. Build → Realtime Database → Create Database
4. Select a region → **Start in test mode** (public read/write rules)
5. Copy the database URL: `https://<project-id>-default-rtdb.firebaseio.com`

Set in `.env`:

```env
CHANNEL=firebase
FIREBASE_URL=https://<project-id>-default-rtdb.firebaseio.com
FIREBASE_INBOX_PATH=c2/inbox
FIREBASE_OUTBOX_PATH=c2/outbox

# Optional but recommended — field name obfuscation
FIREBASE_INBOX_COLUMN_MAP={"command_id":"f3a7k","command":"x9m2p","payload":"b4r8w","target":"d1n5q","status":"h6v3j","created_at":"k2y9t"}
FIREBASE_OUTBOX_COLUMN_MAP={"command_id":"p7c4s","client_id":"m1z8e","status":"w5g2u","result":"a9b3l","timestamp":"r6q7n"}
```

## Database Structure

```
/<FIREBASE_INBOX_PATH>/
  <sha256[:12]>: { command_id, command, payload, target, status, created_at }
  <sha256[:12]>_f0: { ...fragment... }   # fragment entries, if fragmentation is enabled
  <sha256[:12]>_f1: { ...fragment... }

/<FIREBASE_OUTBOX_PATH>/
  <sha256[:12]>: { command_id, client_id, status, result, timestamp }
```

Path keys are the first 12 hex characters of `SHA-256(command_id)` — opaque strings that reveal nothing about the entry type or content. Fragment entries use `{hash}_fN` keys. Field names within each entry are the logical names shown above, or random strings if field name obfuscation is enabled.

Cleanup uses the same hash to reconstruct the exact path key for `DELETE` requests.

---

## Encryption

Firebase stores data as a public JSON document — anyone with the database URL can read it. Fernet encryption is **strongly recommended** for Firebase deployments:

```env
ENCRYPTION_METHOD=fernet
ENCRYPTION_KEY=<key generated by setup wizard>
```

With Fernet enabled, every field value is encrypted before being written to Firebase. An observer with database access sees opaque base64 blobs — without the key, the content is noise.

---

## Field Name Obfuscation

Firebase JSON field names can be replaced with random strings using `FIREBASE_INBOX_COLUMN_MAP` and `FIREBASE_OUTBOX_COLUMN_MAP`. The setup wizard generates and writes these automatically — enabled by default.

Combined with path key hashing and Fernet encryption, an observer with database access sees only opaque hex path keys, random field names, and encrypted values.

```env
FIREBASE_INBOX_COLUMN_MAP={"command_id":"f3a7k","command":"x9m2p","payload":"b4r8w","target":"d1n5q","status":"h6v3j","created_at":"k2y9t"}
FIREBASE_OUTBOX_COLUMN_MAP={"command_id":"p7c4s","client_id":"m1z8e","status":"w5g2u","result":"a9b3l","timestamp":"r6q7n"}
```

Both client and server must use the same maps. Values are encrypted before keys are renamed — on read, keys are restored to logical names before decryption.

---

## Cleanup

When a result arrives from the client, the server automatically deletes both the inbox entry (task) and the outbox entry (result) via `DELETE` requests. Both sides stay lean with no manual intervention.

Fragment entries (`{id}_f0`, `{id}_f1`, ...) are not auto-cleaned. Use the Firebase console → Data → delete the inbox path to clear manually when needed.

---

## Fragmentation

Fragmentation works identically to the Sheets channel. Firebase does not have a field size limit comparable to Google Forms, so `passthrough` (no fragmentation) is sufficient for most payloads. The `fixed` fragmenter is available if needed.

```env
FRAGMENT_METHOD=passthrough   # recommended for Firebase
```

---

## Limitations

- **Fragment cleanup:** Only the primary `{command_id}` inbox and outbox keys are auto-deleted. Fragment keys (`{id}_fN`) remain until manually cleared.
- **No push:** The channel uses polling (same as Sheets). Firebase supports real-time SSE/websocket subscriptions but the current implementation uses REST GET.
- **Public database:** Firebase Realtime Database in test mode is world-readable and world-writable. Always use Fernet encryption in any non-lab environment.
